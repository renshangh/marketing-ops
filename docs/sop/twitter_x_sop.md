# Twitter / X Daily Engagement SOP

This SOP captures the working process that performed well for Mark during live X engagement ops.

It is designed for repeatable daily use inside marketing-ops.

---

## Objective

Build visibility, relevance, and warm audience familiarity on X by replying to high-signal posts without looking spammy, duplicative, or bot-like.

Primary rule:
- **reply to relevant posts like a real operator, not a reply bot**

---

## Core Operating Rules

### Hard rules
- **1 post = 1 reply max**
- **Never duplicate a reply on the same post**
- **Skip ads / sponsored posts**
- **Skip low-confidence UI states**
- **Skip low-signal junk threads** (`nice`, `great share`, `thanks`, etc.)
- **Do not force a batch if the feed quality drops**
- **Short replies beat polished paragraphs**

### Tone rules
Replies should feel:
- short
- human
- casual
- direct
- useful
- not over-explained

Avoid:
- sounding like a polished LinkedIn post
- generic praise with no opinion
- abstract “AI thought leader” language
- repetitive templates
- anything that feels relay/botty/canned

Best target shape:
- **1–2 short sentences**
- simple words
- fast reaction energy
- one clear point

Examples:
- `Yep. This is the real bottleneck.`
- `Exactly. Feels way more intuitive.`
- `This is the part most demos skip.`
- `Strong take. Distribution is the hard part.`

---

## Daily Engagement Goal

Recommended cadence:
- work in **batches of 4**
- prefer **quality over quota**
- if the timeline becomes low-signal, stop or jump to a cleaner section

Suggested daily target:
- **12–30 strong replies/day**
- only if quality remains high

---

## Post Selection Rules

Prioritize posts that are:
- relevant to OpenClaw / agents / workflows / ops / automation / GTM
- posted by builders, operators, technical founders, or adjacent audiences
- likely to benefit from a smart public reply
- strong enough to create recognition through repeated participation

Good categories:
- agent workflows
- browser automation
- AI ops
- orchestration
- memory / context
- GTM systems
- social growth systems
- automation tradeoffs
- workflow/product design

Skip posts that are:
- sponsored / ads
- engagement bait only
- follow-for-follow or empty “say hi” threads
- huge reply piles full of low-value comments
- already replied to by our account
- unclear / unstable in UI state

---

## Execution Flow

### Step 1: Start from a clean timeline state
Before replying:
- make sure the browser is on **Home**
- close stuck composers, modals, or menus
- avoid continuing from messy UI drift

### Step 2: inspect visible posts
For each candidate post:
- confirm it is organic
- confirm it is relevant
- confirm our account has **not** already replied
- confirm the reply button is available

### Step 3: open reply composer
Only continue if:
- the composer clearly shows `Replying to @handle`
- the textbox is available
- the send action is clearly tied to the reply composer

If anything is ambiguous:
- **close it and skip**

### Step 4: write short human reply
Rules:
- 1–2 sentences max
- make one point only
- use natural conversational phrasing
- don’t over-package

### Step 5: verify before send
Before sending, confirm:
- correct target handle/post
- no prior reply from our account in-thread
- actual reply composer is active
- send button is live

## UI Safety Rules

### If the browser gets messy
Common failure states:
- sticky composer/draft remains open
- wrong page/profile opens
- modal overlays block actions
- reply button lags / doesn’t hydrate
- “Post” appears where “Reply” should be

Handling rules:
- if uncertain, **pause and reset**
- return to Home if needed
- do not click blind
- do not post unless the composer is clearly attached to the target post

### If the feed quality collapses
If timeline becomes:
- junk replies
- low-signal comments
- duplicate-looking content
- ambiguous UI clusters

Then:
- stop the batch early
- jump to a cleaner section
- or end the session

---

## Batch Rules

Default batch size:
- **4 at a time**

Meaning:
- select up to 4 unique posts
- verify each individually
- skip uncertainty instead of filling quota blindly

Important:
- a “batch of 4” is a ceiling, **not a quota requirement**
- if only 2–3 clean posts are available, post only 2–3

---

## Reply Quality Checklist

A reply is good if it is:
- relevant
- short
- human
- specific
- non-duplicative
- easy to read fast

A reply is bad if it is:
- too long
- too polished
- too generic
- trying too hard to sound smart
- basically reusable on any post

---

## Suggested Daily Workflow

### Morning loop
1. open X Home
2. scan timeline for strong organic posts
3. reply to up to 4 unique posts
4. confirm sends
5. repeat only if quality remains high

### During operation
- keep running total if needed
- note what kinds of posts get good traction
- adjust tone toward shorter/more human replies if needed

### End of session
Capture:
- number of unique replies sent
- what post types worked best
- what failed in browser/UI
- any new operating rules worth preserving

## Recommended Reply Formula

Use one of these simple patterns:

### Pattern A: agree + one sharp point
- `Yep. This is the real bottleneck.`
- `Exactly. The workflow part is what matters.`

### Pattern B: quick reaction + operator lens
- `This is underrated. Most teams miss this part.`
- `Feels true. The edge is in execution.`

### Pattern C: contrast
- `The demo is easy. Reliable ops are the hard part.`
- `Ideas are cheap. Clean systems are harder.`

### Pattern D: process framing
- `Exactly. It’s a process game.`
- `This is really a workflow problem.`

---

## What Not To Do

Do not:
- reply to the same post twice
- mass-reply to junk threads
- use long, essay-style replies by default
- trust the browser state without checking
- force a batch when the feed gets weak
- post if the UI looks ambiguous
- confuse a standalone post composer with a reply composer

---

## Recommended V1 Standard

For daily marketing ops execution, use this standard:

- **short human replies**
- **4-post batches**
- **organic posts only**
- **strict no-duplicate rule**
- **skip uncertainty**
- **verify composer before send**

That is the baseline operating model unless a better one replaces it.
